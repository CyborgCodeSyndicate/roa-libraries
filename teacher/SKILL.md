# AI Teacher — Code Generation from Lesson Catalog

Before writing any new Java code, consult the lesson catalog generated by the `ai-teacher-plugin` to find curated examples that match what is being built.

## Step 1 — Resolve the lesson catalog directory

Look for lesson JSON files in (in priority order):
1. `target/pandora/ai-teacher/` relative to the module being modified
2. `<project-root>/target/pandora/ai-teacher/` as a fallback

**If the directory does not exist or contains no JSON files**, run the Maven plugin to generate the catalog:

```bash
mvn pandora:teach
```

The plugin requires `<packages>` to be configured in the consuming project's `pom.xml` under the `ai-teacher-plugin` configuration. If the command fails because of missing configuration, inform the user and ask them to configure the plugin before proceeding. Do not block code generation indefinitely — if the catalog cannot be produced, proceed with the available context.

## Step 2 — Pick the category file

Use the **Category Reference** table below to decide which JSON file to open based on what is being generated. Open only the file(s) that match — do not load the entire catalog.

### Category Reference

| What is being generated | Category file |
|---|---|
| Service class / method | `service.json` |
| Repository / DAO | `repository.json` |
| Controller / REST endpoint | `controller.json` |
| Test class / method | `test.json` |
| Configuration class | `config.json` |
| Utility / helper class | `util.json` |
| DTO / request / response object | `dto.json` |
| Entity / domain model | `model.json` |
| Filter / interceptor / middleware | `filter.json` |
| Event / listener / handler | `event.json` |

If the project contains a category file not listed here, use the closest semantic match. If none apply, proceed without a catalog.

## Step 3 — Select the right examples using tags

After opening the category file, use the **Tag Reference** table below to understand what each tag signals. Compare the tags on each catalog item against the current generation context — what kind of logic, pattern, or concern is being implemented — and select the items whose tags best fit.

### Tag Reference

| Tag | What it signals — use items with this tag when… |
|---|---|
| `api` | Code makes HTTP calls or interacts with an external API |
| `assertions` | Code validates or asserts response content, field values, or state |
| `auth` | Code handles authentication, tokens, login, or session management |
| `chain` | Code composes multiple calls or steps in a sequential pipeline |
| `config` | Code defines or reads application configuration / beans |
| `create` | Code performs a create / POST operation |
| `delete` | Code performs a delete / DELETE operation |
| `dto` | Code maps, deserializes, or constructs data transfer objects |
| `fluent` | Code uses a fluent / builder style API |
| `get` | Code performs a read / GET operation |
| `header` | Code sets, reads, or validates HTTP headers |
| `integration` | Code is an integration test touching real infrastructure |
| `mock` | Code uses mocks or stubs to isolate units under test |
| `pagination` | Code handles paged responses, cursors, or list metadata |
| `retry` | Code applies retry logic or resilience patterns |
| `security` | Code enforces authorization rules or security constraints |
| `spring` | Code relies on Spring beans, annotations, or context |
| `timestamp` | Code generates, stores, or asserts date/time fields |
| `unit` | Code is a focused unit test with no external dependencies |
| `update` | Code performs an update / PUT / PATCH operation |
| `validation` | Code validates request input or response structure |

Tags in catalog items are comma-separated within a single string — split on `,` before matching.

Prefer `EXCELLENT` items over `GOOD`. Never use `BAD` items as generation references — cite them only to warn the user about anti-patterns to avoid.

Select up to **3 most relevant items** to keep context focused.

## Step 4 — Read the example source files

For each selected item:
- Read the file at the `path` field (relative to the project root).
- Read each entry in the item's `related` array as well, to understand the full call chain.

Skip gracefully if a file does not exist on disk.

## Step 5 — Generate the code

Write the new code using the selected items as the reference pattern:
- Follow the structural approach demonstrated in the example sources.
- Apply the `whenToUse` and `description` guidance from the matched items literally.
- Reference the same collaborator classes found in `related` where they fit the new context.
- Adapt — do not copy verbatim.

After the code block, cite which catalog item(s) (id + category) informed the output.

## Example workflow

User asks: *"Write a service method that calls the payment API and validates the response."*

1. Check `target/pandora/ai-teacher/` — if empty, run `mvn pandora:teach`
2. Category table → open `service.json`
3. Tag table: context involves `api`, `validation`, `assertions` → select items whose tags overlap most; prefer `EXCELLENT`
4. Read the source at each matched item's `path`; read any `related` files
5. Generate the method following the same style shown in the examples
6. Cite the reference item(s)

## Notes

- The catalog is authoritative for this project's patterns — treat it like a curated code review from a senior engineer.
- If the user has already shared relevant code in the conversation, that takes precedence over catalog examples with fewer matching tags.
- The `@AiLesson` annotation is placed exclusively by the human developer — never add it to generated code.
